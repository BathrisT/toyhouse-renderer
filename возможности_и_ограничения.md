# Toyhou.se: возможности и ограничения кастомного HTML в дизайнах профилей

Гайд собран по результатам построчного разбора 14 реальных примеров дизайнов из
`examples_from_toyhouse/` (`1.html … 12.html`, `13_folders.html`, `14.html`). Ниже —
только то, что **реально встречается** в живых, рабочих шаблонах, плюс явные выводы о
том, чего в них принципиально нет. Если тег/приём ни разу не встретился ни у одного из
разных независимых авторов — считай его непроверенным и не полагайся на него, пока не
протестируешь вручную на сайте.

## ⚠️ КРИТИЧНО #1: WYSIWYG обязательно выключать — проверено эмпирически на живой странице

Это уже не пересказ чужих предупреждений из комментариев — сделан прямой тест на
реальном toyhou.se: один и тот же HTML вставлен дважды (с включённым и с выключенным
WYSIWYG), результаты сохранения сверены посимвольно.

**Если WYSIWYG включён**, редактор Froala (WYSIWYG-движок Toyhouse) при сохранении
уродует контент так:
- **Все `<i class="fa-...">`-иконки удаляются полностью**, без остатка — ни тега, ни
  класса нигде не остаётся (проверено: на реальной странице было 0 из ~50 иконок).
  Похоже, Froala чистит "пустые" инлайновые теги форматирования — у `<i>` нет
  текстового содержимого, только CSS-иконка через `::before`, и редактор считает его
  мусором от редактирования.
- **У `<img>` `width`/`height` из `style=""` переезжают в нативные HTML-атрибуты
  `width=""`/`height=""`, причём часто СО ЗНАЧЕНИЕМ РЕАЛЬНОГО ОТРЕНДЕРЕННОГО РАЗМЕРА НА
  МОМЕНТ СОХРАНЕНИЯ, а не тем, что было в коде** (в тесте: задавал 104×104 → сохранилось
  74×74 в одном прогоне, 86×86 в другом). Картинка внутри вкладки, которая была скрыта
  (не активна) в момент сохранения — рискует получить `width="0" height="0"`, то есть
  визуально исчезнуть насовсем. Все выжившие `<img>` дополнительно получают классы
  `fr-fil fr-dib` — это внутренние классы Froala (тот же движок требует классы
  `fr-video fr-fvc fr-dvi fr-draggable` на `<video>`, см. 3.7 — теперь понятно, откуда
  это берётся).
- **Пустые декоративные `<div></div>` не удаляются, но внутрь молча вставляется `<br>`.**
  Обычно не критично визуально (у таких div фиксированный размер через style), но
  искажает разметку.

**Если WYSIWYG выключён** (поле ввода переведено в режим исходного кода) — всё
сохраняется **байт-в-байт как в оригинале**: иконки все на месте, `<img>` не тронуты,
`<br>` в пустые div не вставляются.

**Вывод:** без выключения WYSIWYG любой дизайн, полагающийся на `<i>`-иконки Font Awesome
или на точные размеры `<img>`, будет визуально сломан на сохранённой странице — даже если
сам HTML написан абсолютно правильно. Это не "рекомендация", а обязательный шаг.

## ⚠️ КРИТИЧНО #2: серверный санитайзер стилей режет отдельные CSS-свойства и ломает сложные shorthand — даже с выключенным WYSIWYG

Это отдельная находка, всплывающая **после** решения проблемы №1 — то есть работает
независимо от режима WYSIWYG. Судя по всему, при сохранении на сервере отдельно (помимо
Froala) прогоняется санитайзер именно для `style=""`, который парсит CSS и пересобирает
его — и на части синтаксиса откровенно ломается.

**1) `gap` / `row-gap` / `column-gap` в `style=""` не выживают вообще** — молча
вырезаются, при этом остальные свойства в том же `style=""` сохраняются:
```html
<!-- было -->
<div class="d-flex flex-wrap mt-3" style="gap:8px 18px; font-size:12.5px; color:var(--mute);">
<!-- стало после сохранения -->
<div class="d-flex flex-wrap mt-3" style="font-size:12.5px;color:var(--mute);">
```
Проверено на живой странице: 0 из ~15 использований `gap`/`row-gap` пережили сохранение.
Похоже, allowlist разрешённых CSS-свойств для `style=""` не включает эти относительно
новые для flexbox-контекста свойства (изначально `gap` был только для grid).

**2) Сложный многослойный `background:` shorthand (градиент + картинка + position/size
в одном объявлении) пересобирается в мусор**, теряя часть слоёв:
```html
<!-- было -->
style="background:linear-gradient(0deg, var(--panel) 0%, rgba(3,5,10,0) 55%), url('...') center/cover;"
<!-- стало после сохранения -->
style="background:url(...) var(--panel) var(--panel);"
```
Градиент-слой и `center/cover` полностью потеряны, вместо них — `var(--panel)`,
подставленный дважды подряд как попало (не валидный `background-position`/`-size`).
Санитайзер, похоже, парсит `background` как CSSOM-объект и пересериализует его обратно
в текст, и на многослойных фонах с `linear-gradient()` + `url()` вместе спотыкается.

Заодно санитайзер минифицирует весь `style=""` (убирает пробелы после `:`/`;`) — само по
себе не проблема, но подтверждает, что стиль реально проходит отдельную обработку,
отличную от обработки тегов/атрибутов Froala.

**Практические выводы для генерации новых шаблонов:**
- Не полагаться на `gap`/`row-gap`/`column-gap` для отступов между flex/grid-элементами.
  Вместо этого:
  - если нужный отступ совпадает со шкалой Bootstrap (4/8/16/24/48px) — использовать
    **классы** `mr-*`/`mb-*`/`mx-*`/`my-*` и т.п. (это `class=""`, не `style=""` —
    под санитайзер стилей не попадает в принципе);
  - если нужно произвольное значение — ставить `margin-right`/`margin-bottom` явно на
    **каждый** дочерний элемент (классическая до-`gap` техника для flex), а не `gap` на
    родителе;
  - для вертикального интервала между обёртками Bootstrap `.row`/`.col` (`row-gap` тоже
    не выживает) — вешать `mb-*`-класс на сами `.col`, а не `row-gap` на `.row`.
- Не комбинировать `linear-gradient()` + `url()` + `position/size` в одном свойстве
  `background:`. Разбивать на отдельные простые, однозначные объявления
  (`background-color:`, `background-image:`, `background-position:`,
  `background-size:`) — по одному значению на свойство, без запятых/составных
  shorthand-конструкций. Если нужен эффект "картинка + градиентная плашка поверх" —
  надёжнее сделать это **двумя отдельными слоями** (например, картинка как
  `background-image` на одном `<div>`, а градиент — как `background: linear-gradient(...)`
  (без url) на отдельном абсолютно позиционированном `<div>` поверх), чем пытаться
  засунуть всё в одно многослойное CSS-значение.
- Общий принцип: чем сложнее/составнее CSS-значение в `style=""` (несколько слоёв через
  запятую, смешанные shorthand с разнотипными частями), тем выше риск, что санитайзер
  его пересоберёт неправильно. Простые, однозначные, одно-значные объявления — надёжнее.

## ⚠️ КРИТИЧНО #3: `<div>` (блочный элемент) внутри `<a>` — санитайзер разрывает `<a>` на куски

Тоже проявляется независимо от WYSIWYG. По спецификации HTML5 `<a>` спокойно может
оборачивать блочные элементы (`<div>` внутри `<a>` — валидно, так уже много лет), но
санитайзер Toyhouse, похоже, следует более старой HTML4-логике, где `<a>` — строчный
(inline) элемент и не может содержать блочные потомки. Он не удаляет вложенный `<div>`,
а **"чинит" разметку, разрывая один `<a>` на множество обрывков** — закрывает `<a>` перед
каждым `<div>`, вставляет `<div>` уже СНАРУЖИ, и открывает новый `<a>` с теми же
атрибутами для следующего кусочка контента. На практике карточка вида:

```html
<a href="LINK" class="d-flex align-items-center p-3" style="...">
  <img src="...">
  <div>
    <div>Имя</div>
    <div>Подпись</div>
  </div>
  <i class="..."></i>
</a>
```

превращается в кашу из полудюжины отдельных `<a href="LINK" class="..." style="...">`,
каждый из которых оборачивает только один соседний кусок, а `<div>`-ы вылезают между
ними — визуально и структурно всё ломается (двойные рамки/фоны на каждом обрывке,
непредсказуемая кликабельность).

**Обход — тот же приём, что уже описан в 3.8 (`stretched-link`):**
```html
<div class="d-flex align-items-center p-3" style="position:relative; background:...; border:...;">
  <img src="...">
  <div>
    <div>Имя</div>
    <div>Подпись</div>
  </div>
  <i class="..."></i>
  <a href="LINK" class="stretched-link"></a>
</div>
```
Весь видимый контент (картинки, текстовые `<div>`, иконки) переносится в обычный
`<div>`-контейнер (`position:relative`), а кликабельность всей карточки обеспечивает
**пустая** `<a class="stretched-link"></a>` в конце — у неё нет вложенных блочных детей,
санитайзеру нечего "чинить". Если нужен тултип на всю карточку — вешать
`data-toggle="tooltip"`/`title` тоже на эту пустую `<a>`, а не на обёртку.

**Практический вывод:** если карточка/плашка должна одновременно (а) быть кликабельной
целиком и (б) содержать `<div>`-ы внутри (не просто текст/иконки) — не оборачивать всё в
`<a>`. Использовать `stretched-link` на пустой `<a>` внутри позиционированного `<div>`.
Обычные `<a>` с ТОЛЬКО инлайновым содержимым (текст, `<i>`, `<span>`) — по-прежнему
безопасны, проблема именно в `<div>`-потомках.

## ⚠️ КРИТИЧНО #4: собранный на Toyhouse Bootstrap CSS — не то же самое, что CDN-версия. `.collapse` слабее, чем в стоке, и конфликтует с `.nav`/`.d-flex` на одном элементе

Это не баг санитайзера (тот вообще не участвует) — это отличие в том, **как на самом
Toyhouse скомпилирован их собственный `site_bootstrap.css`**, от того, что грузится с
CDN (`cdn.jsdelivr.net/npm/bootstrap@4.6.2`), которым я тестировал вёрстку локально
перед вставкой. Из-за этого локальное превью может выглядеть и вести себя корректно, а
на живом toyhou.se — сломанно, причём именно в этом единственном месте разница и
всплывает.

**Конкретика:** в стоковом Bootstrap 4 правило скрытия collapse-элемента —
```css
.collapse:not(.show) { display: none; }   /* специфичность 0,2,0 — специально высокая,
                                              чтобы гарантированно перебивать другие
                                              классы display на том же элементе */
```
На Toyhouse же (проверено напрямую через `document.styleSheets` на живой странице) это
правило скомпилировано проще:
```css
.collapse { display: none; }              /* специфичность 0,1,0 — как у любого
                                              одиночного класса */
```
Из-за этого, если на ОДНОМ элементе одновременно висят `.collapse` и любой другой класс,
тоже задающий `display` (`.nav` → `display:flex`, `.d-flex`, `.d-grid` и т.п.), решает
уже не "кто специфичнее", а "кто в файле объявлен позже" (правило CSS для равной
специфичности) — а `.nav` в файле Toyhouse идёt позже `.collapse`. Итог: элемент с
`class="nav ... collapse"` **никогда не скрывается**, независимо от того, есть на нём
`.show` или нет — коллапс визуально не работает вообще (ни "закрыт по умолчанию", ни
"закрывается по повторному клику").

Проверено практически: `<ul class="nav collapse">` → всегда `display:flex` на реальном
Toyhouse. `<div class="collapse">` (без `.nav`/`.d-flex` рядом) → корректно `display:none`
по умолчанию и честно переключается через `data-toggle="collapse"`.

**Практический вывод:** никогда не вешать `.collapse` на тот же элемент, что и `.nav`,
`.d-flex`, `.d-grid` или любой другой класс, задающий `display` напрямую. Разносить на
два элемента — внешний с ТОЛЬКО `.collapse` (+ при необходимости `d-md-block`/`d-md-none`
для десктопной версии, эти классы используют `!important` и с `.collapse` не
конфликтуют), внутренний — с `.nav`/`.d-flex`/т.п.:
```html
<div class="collapse d-md-block" id="mobileNav">
  <ul class="nav nav-pills flex-column">...</ul>
</div>
```
а не
```html
<!-- НЕ ТАК: -->
<ul class="nav nav-pills flex-column collapse d-md-flex" id="mobileNav">...</ul>
```

**Более широкий вывод:** локальный CDN-превью (см. `renders/`) — надёжный способ
проверить общую вёрстку/логику, но **не гарантия точного совпадения с живым Toyhouse** —
их сборка Bootstrap может отличаться в деталях специфичности CSS-правил. Финальную
проверку конкретно интерактивных Bootstrap-компонентов (collapse, tabs, carousel,
tooltip) стоит по возможности делать на самой площадке, а не только локально.

## 0. Что это вообще за HTML

- Это **не полные документы**. Ни в одном файле нет `<!DOCTYPE>`, `<html>`, `<head>`,
  `<body>` — только фрагмент разметки, который вставляется в текстовое поле профиля/
  персонажа на toyhou.se (bio, "literature", custom info-поля и т.п.).
- Поле ввода — WYSIWYG-редактор (Froala). Перед вставкой кода обязательно **выключить
  WYSIWYG-режим** — см. «КРИТИЧНО #1» выше за подробностями, что именно ломается, если
  забыть это сделать.
- HTML-**комментарии** (`<!-- -->`) авторы используют очень активно для структурирования
  шаблона и инструкций пользователю, но несколько шаблонов прямо указывают: **сайт
  вырезает комментарии при сохранении** — они видны только в "сыром" черновике. Поэтому
  для удобного редактирования сообщество рекомендует внешние live-превью редакторы
  (circlejourney.net, af.circlejourney.net), а не сам сайт.
- Парсер **не строгий, но и не всепрощающий**: регистр атрибутов не важен (`STYLE=`
  работает так же, как `style=`), но опечатка в имени класса (`contianer` вместо
  `container`) просто не даёт эффекта — без ошибок, но и без стилизации.
- Часть шаблонов кросс-постится и на artfight.net — иногда встречаются комментарии
  "если используешь на Toyhouse, а не Artfight — убери этот divider". Это нюанс
  происхождения шаблона, а не ограничение самого Toyhouse.

## 1. Чего категорически НЕТ ни в одном из примеров

Проверено полным чтением всех 14 файлов + grep по ним сразу — совпадений **ноль**:

- `<style>` — блоков стилей нет вообще. **Вся стилизация — только через инлайновый
  атрибут `style="..."`.**
- `<script>` — никакого JS от пользователя.
- `@keyframes`, `@font-face`, `@import` — нельзя объявить свою анимацию или подключить
  свой шрифт/внешний CSS.
- `on*="..."` (onclick, onmouseover и т.д.) — обработчики событий отсутствуют.
- `<canvas>`, `<svg>`, `<marquee>`, `<details>`/`<summary>`, `<form>`/`<input>` (в т.ч.
  checkbox/radio-хаки для вкладок), `<button>` (вместо него используют `<a>`/`<div>` с
  классом `btn`).

**Вывод:** вся "интерактивность" в дизайнах toyhou.se — это не пользовательский JS, а
готовые JS-плагины, которые **уже подключены самим сайтом** (Bootstrap JS + jQuery) и
реагируют на служебные `data-*`-атрибуты. Пользователь не пишет логику — он просто
использует правильную разметку, а сайт её "оживляет".

## 2. Что уже подключено сайтом (доступные "кубики")

- **Bootstrap 4.x** — сетка, утилиты, компоненты (`card`, `nav`, `tab-content`,
  `collapse`, `carousel`, `progress`, `badge`...). Синтаксис `data-toggle="tab"` (а не
  `data-bs-toggle`) подтверждает именно Bootstrap 4, не 5.
- **jQuery** — неявно, обеспечивает работу Bootstrap-плагинов.
- **Font Awesome**, судя по всему довольно полный/Pro-набор: встречаются и классические
  префиксы (`fas`, `far`, `fal`, `fad`, `fab`, `fat` — Thin), и синтаксис FA6/7 (`fa-solid`,
  `fa-regular`, `fa-light`, `fa-duotone`, `fa-brands` рядом с именем иконки), и Sharp-стиль
  (`fass`, `fast`, `fa-sharp`). **Проверено напрямую на живой странице:** подключённый
  шрифт — `font-family: "Font Awesome 7 Brands"` (т.е. на сайте реально **Font Awesome 7**,
  не 6 — если ориентироваться на актуальный список иконок, брать за основу версию 7, хотя
  большинство используемых имён иконок общие для 6 и 7). Плюс встроенные анимационные
  модификаторы FA6/7:
  `fa-spin`, `fa-spin-pulse`, `fa-beat`, `fa-fade`, `fa-shake`, `fa-flip-horizontal`
  (упоминается и `fa-flip`, `fa-bounce` как альтернативы) — это **готовые CSS-анимации из
  самого FA**, их можно использовать, хотя `@keyframes` самому объявить нельзя.
  Подтверждён и родной **FA Stack** (иконка-в-круге/бейдж), рабочий пример из `14.html`:
  ```html
  <span class="fa-stack fa-lg">
    <i class="fa-solid fa-circle fa-stack-2x" style="color:#1b294f"></i>
    <i class="fat fa-stars fa-stack-1x" style="color:#c4cfe0"></i>
  </span>
  ```
  Также встречается `fa-inverse` (инвертированный цвет поверх заливки) и мелкоступенчатая
  шкала размеров FA6 (`fa-2xs`, `fa-xs`, `fa-sm`, `fa-lg`, `fa-2xl`...) вместо/вместе со
  старой `fa-{1..10}x`.
- **Tooltipster** (jQuery-плагин тултипов) — используется наравне с нативным тултипом
  Bootstrap, см. раздел 3.4.
- Собственный, не-стоковый класс сайта: **`bg-faded`** — встречается в нескольких
  независимых шаблонах, значит это часть кастомной Bootstrap-сборки Toyhouse.

## 3. Рабочие интерактивные паттерны (проверенные, готовые к копированию)

### 3.1 Вкладки (Tabs)

```html
<ul class="nav nav-tabs">
  <li class="nav-item"><a class="nav-link active" data-toggle="tab" href="#info">INFO</a></li>
  <li class="nav-item"><a class="nav-link" data-toggle="tab" href="#persona">PERSONA</a></li>
</ul>
<div class="tab-content">
  <div class="tab-pane fade show active" id="info">...</div>
  <div class="tab-pane fade" id="persona">...</div>
</div>
```

- Первая `tab-pane` помечается `active` (и обычно `show`); в паре шаблонов ещё встречается
  устаревший BS3-класс `in` рядом с `show` — лишний, но безвредный.
- Важно: `nav`/`nav-tabs`-обёртка и `tab-content` **не обязаны быть соседями в DOM** — в
  `1.html` навигация "cast" находится в одной колонке, а её `tab-content`/`tab-pane` — в
  совершенно другом блоке страницы. Работает всё равно, важно только совпадение
  `href="#id"` ↔ `id="id"`.
- `aria-*`/`role` атрибуты не обязательны — в реальных шаблонах их почти никогда нет.

### 3.2 Аккордеон (Collapse)

```html
<div class="card">
  <div class="card-header">
    <a data-toggle="collapse" href="#personalitySection" data-parent="#personaAccordion">
      PERSONALITY TRAITS
    </a>
  </div>
  <div id="personalitySection" class="collapse show">
    <div class="card-body">...</div>
  </div>
</div>
```

- `data-parent="#accordionId"` на всех триггерах одной группы даёт классическое
  "открыта только одна секция за раз".
- `aria-expanded="true"` иногда ставится только на первый триггер, а на остальные — нет
  (несогласованность у самих авторов, но на работу не влияет).
- Тот же `data-toggle="collapse"` + `data-target="#id"` используется и **вне** карточек
  (например, в `8.html` — сворачиваемые блоки треков плейлиста, где триггер и `.collapse`
  лежат в разных "половинах" одной строки).

### 3.3 Карусель (Carousel)

```html
<div class="carousel slide" id="appearance" data-ride="false" data-pause="true">
  <div class="carousel-inner">
    <div class="carousel-item active">...</div>
    <div class="carousel-item">...</div>
  </div>
</div>
<a href="#appearance" data-slide="prev"><i class="fas fa-angles-left"></i></a>
<a href="#appearance" data-slide="next"><i class="fas fa-angles-right"></i></a>
```

- Стрелки не обязаны иметь стандартные классы `carousel-control-prev/next` — можно
  полностью кастомно оформленные `<a>` со своими `data-slide`, лишь бы `href` указывал на
  `id` карусели.
- `data-ride="false"` — чтобы не листалась сама.

### 3.4 Тултипы

Встречаются **два разных** механизма, оба используются в реальных шаблонах:

```html
<!-- Bootstrap native -->
<a data-toggle="tooltip" title="#000000">...</a>

<!-- Tooltipster (jQuery-плагин) -->
<a class="tooltipster" data-placement="right" title="Code: PinkyDoggy">...</a>
```

Второй вариант чаще всего используется именно для обязательной "кредитки" авторства кода.
`data-toggle="tooltip"` в `14.html` подтверждён на **любых** элементах, не только `<a>`:
на `<span>` (например, для ника в Discord, у которого нет ссылки), на `<img>`
(`<img ... data-toggle="tooltip" title="username">` — подпись при наведении на аватар
друга), и на `<div class="card">` целиком (подсказка про статус коммишенов).

### 3.5 Скрытая фоновая музыка через YouTube-iframe (самый частый трюк)

```html
<iframe src="https://www.youtube.com/embed/VIDEOID?controls=0"
        frameborder="0"
        style="opacity:.0001; position:absolute; overflow:hidden;"></iframe>
```

- Встречается в 4 из 12 файлов (`1.html`, `4.html`, `7.html`, `8.html`), почти дословно
  одинаковый приём: `iframe` делают почти невидимым (`opacity` в районе `0.0001–0.001`),
  часто `position:absolute` + `overflow:hidden`/маленький `transform: translate(...)`, и
  накладывают его поверх декоративной кнопки play (обычно иконка FA). Есть вариант с
  `youtube-nocookie.com` вместо `youtube.com`.
- `?controls=0` в URL — единственный "параметр" в этих примерах; автозапуска через
  query-параметры (`autoplay=1`) в примерах не наблюдалось.
- Есть и минималистичный вариант скрытия (`14.html`) — без обёртки в `<span>`, совсем
  крошечный `iframe` вместо процентного растягивания на весь блок:
  ```html
  <iframe frameborder="0" style="position:absolute;height:1rem;width:1rem;opacity:0.001"
          src="https://www.youtube.com/embed/VIDEO_ID"></iframe>
  ```
  Оба варианта (большой на весь блок с `overflow:hidden` и крошечный `1rem×1rem`) реально
  встречаются у независимых авторов — значит размер/обёртка не принципиальны, важна только
  связка `opacity≈0` + `position:absolute`.

### 3.6 Настоящий `<audio>`

`10.html` — по сути готовый čheat-sheet вариантов плеера, **не** цельный дизайн. Важные
факты именно из него (в т.ч. прямо прописаны в тексте шаблона как ограничения площадки):

- Работают только **прямые ссылки на файл** (`.mp3`). YouTube через `<audio>` **не
  работает** — только через iframe-трюк (3.5).
- Рекомендованные хостинги: File Garden, SndUp, Dropbox (прямая ссылка), Vocaroo.
  Google Drive и Discord CDN помечены как ненадёжные (ссылки протухают).
- Автор прямо пишет: сайт "**блокирует часть возможностей кода**" — не работают
  `autoplay`, задание стартовой громкости и плейлисты (то есть подобные атрибуты либо
  вырезаются санитайзером, либо игнорируются браузером в этом контексте).
- `#t=10` в конце URL файла — устанавливает точку старта воспроизведения.
- Каталог визуальных вариаций плеера, все — чистый inline CSS:
  - "призрачный" плеер: `opacity: 0.2–0.99` на `<audio>`;
  - цветная таблетка/квадрат: `background`, `padding`, `border-radius` на `<audio>`;
  - обложка трека: `background-image:url(...); background-size:cover` прямо на `<audio>`;
  - декоративная "полоса прогресса": обёртка классами Bootstrap
    `progress`/`progress-bar`/`progress-bar-striped`/`progress-bar-animated` — это чужая
    CSS-анимация из бандла Bootstrap, а не самописный `@keyframes`;
  - цветные фильтры через `mix-blend-mode` (автор предупреждает — рендерится
    непредсказуемо в разных браузерах);
  - плавающий плеер: класс `fixed-bottom`/`fixed-top`;
  - центрирование через `<table style="width:0%">` + `<td style="width:100%; ...">`
    (классический html-хак "сжаться по контенту и отцентрироваться").

### 3.7 `<video>`

Работает, но только если обёрнуто в служебные классы редактора Froala, иначе не
принимается:

```html
<span class="fr-video fr-fvc fr-dvi fr-draggable" contenteditable="false">
  <video class="fr-fvc fr-dvi fr-draggable" controls width="250">
    <source src="MP4LINKHERE" type="video/mp4">
  </video>
</span>
```

Это утечка внутренней механики WYSIWYG-редактора сайта (Froala) — копировать точь-в-точь,
не менять классы/атрибут `contenteditable="false"`.

### 3.8 `stretched-link` — вся карточка кликабельна одной ссылкой

Подтверждённый рабочий Bootstrap 4.3+ утилитный класс (`14.html`): делает так, чтобы вся
область позиционированного родителя (например, `card`) реагировала на клик, хотя
физически `<a>` — это просто текст внутри:

```html
<div class="card border-0" style="position:relative;">
  <a class="stretched-link" href="LINK">
    <p>commissions</p>
  </a>
</div>
```

Удобно для "кнопок-плашек" (папки, статусы, ссылки на соцсети), где вся карточка должна
быть кликабельной, а не только текст.

### 3.9 Виджет "папки" (сетка кликабельных иконок-разделов)

Отдельный частый тип дизайна (`13_folders.html`) — не карточка персонажа, а **навигационная
сетка** из плиток-"папок" (обычно ссылки на подгалереи toyhou.se). Паттерн одной плитки:

```html
<div class="card bg-faded p-1 mb-2 col-md-2 col-12" style="box-shadow: 4px 4px 6px rgba(0,0,0,.06);">
  <div class="card p-1" style="background-image:url('IMAGEURL'); height:76px; background-position:center;">
    <div class="card border-0 p-3 mb-0 text-center" style="background: rgba(0,0,0,0.3);">
      <a href="FOLDER URL" style="color:#FFFFFF;">
        <i class="fas fa-star fa-2x"></i>
        <p class="font-italic">Folder Name</p>
      </a>
    </div>
  </div>
</div>
```

Фоновая картинка + тёмная `rgba`-плашка поверх + иконка FA + подпись, всё внутри `<a>`.
Автор шаблона прямо рекомендует: **не больше 5 плиток в ряд** (при `col-md-2`, т.к. с
учётом отступов 6 в ряд визуально не помещаются аккуратно), большие/качественные исходные
изображения, и подгонять `background-size` (по умолчанию `cover`, но если не помещается —
использовать явный процент, например `background-size: 80%`, и подбирать число).

### 3.10 Реальные "папки" Тойхауза — что кастомизируется бесплатно, а что за Premium

Не проверено вживую (найдено по форумам toyhou.se, не собственным тестом на реальной
странице — в отличие от разделов КРИТИЧНО выше), но источники сходятся на одной картине:

- У папки (группировки персонажей) есть **своё поле "Folder Description"** — отдельное
  от био персонажа, но по механике идентичное: обычный HTML-фрагмент, те же правила
  (никаких `<script>/<style>`, только инлайн `style=""`), и **то же требование выключить
  WYSIWYG** перед вставкой (КРИТИЧНО #1 актуален и здесь). Именно сюда вставляется виджет
  из 3.9 — доступно **без Premium**.
- Кастомизация/скрытие **самого нативного списка персонажей внутри папки** (стандартной
  Тойхаузовской сетки карточек — заголовок папки, "папка пуста", кнопки сортировки/тегов)
  — через отдельное поле **"Folder CSS"**, которое принимает уже настоящий `<style>`-CSS
  (не только инлайн), нацеленный на встроенные классы папки. По имеющимся сообщениям это
  **требует Premium**-аккаунт (как и общий Profile CSS на весь профиль).
- Отсюда и сам паттерн "виджет-сетка папок" в 3.9: без Premium нельзя переоформить
  штатный список персонажей CSS-ом, поэтому сообщество вместо этого **полностью подменяет
  вид собственным HTML-блоком** в поле описания папки — виджет живёт рядом с (или вместо
  видимой части) нативным списком, а не переопределяет его стили.

Практический вывод для `templates/folders.html`: наш подход (кастомный HTML-виджет в
поле описания, без затрагивания нативных классов папки) — единственный вариант,
работающий **без Premium**, и именно поэтому он собран как самостоятельный виджет, а не
как CSS-переопределение чего-то нативного.

## 4. Сетка и утилиты Bootstrap 4, реально встреченные

- **Контейнеры/сетка:** `container`, `row`, `no-gutters`, `col`, `col-auto`,
  `col-{1..12}`, `col-md-*`, `col-lg-*`, `col-sm-*`.
- **Flex:** `d-flex`, `d-block`, `d-lg-flex`, `flex-column`, `flex-row`, `flex-wrap`,
  `flex-fill`, `flex-grow-1`, `flex-shrink-1`, `justify-content-{center|between|around|end}`,
  `align-items-center`, `order-*`.
- **Отступы**, включая **отрицательные** margin-утилиты Bootstrap 4.1+: `p-*`, `m-*`,
  `px-*`, `py-*`, `mx-*`, `my-*`, `mt-n5`, `ml-n3`, `mb-n2` и т.п. — активно используются
  для "наезжающих" друг на друга декоративных карточек/рамок.
- **Текст:** `text-center`, `text-left`, `text-right`, `text-justify`, `text-truncate`,
  `text-uppercase`, `text-lowercase`, `font-weight-normal`, `font-italic`,
  `display-{2|4}`, `small`.
- **Цветные текстовые/кнопочные утилиты, привязанные к теме сайта:**
  `text-primary/success/warning/danger/info/muted/white`,
  `btn`, `btn-primary/secondary/default` — их итоговый цвет определяется **темой
  оформления конкретного пользователя toyhou.se**, а не жёстко зашитым значением. Это
  удобно, если хочется, чтобы элемент "подхватывал" тему владельца профиля, но
  непредсказуемо, если нужен строго фиксированный цвет — тогда используй inline `style`.
- **Карточки:** `card`, `card-header`, `card-body`, `card-deck`, `card-img`. Также
  встречается **устаревшее** `card-block` (`13_folders.html`) — так называлась `card-body`
  в ранних альфа-версиях Bootstrap 4; разные авторы шаблонов явно сидят на разных
  под-версиях BS4, но обе формы не ломают разметку.
- **Прочие компоненты:** `badge`, `badge-pill`, `progress`, `progress-bar`,
  `progress-bar-striped`, `progress-bar-animated`, `stretched-link` (см. 3.8 — делает всю
  карточку кликабельной одной ссылкой).
- **Позиционирование:** `sticky-top`, `fixed-top`, `fixed-bottom`, `position-relative`,
  `position-absolute`.
- **Скругления/рамки:** `rounded`, `rounded-0`, `rounded-circle`, `rounded-top`,
  `rounded-bottom`, `border-0`, `border-top-0` и т.п.
- **Font Awesome layout-утилита:** `fa-ul`/`fa-li` — списки с иконками вместо стандартных
  буллитов.
- **Замечены, но нестандартны/сомнительны** (встречаются не во всех, возможно наследие
  старых версий Bootstrap и могут не иметь эффекта): класс `in` рядом с `show` на
  `tab-pane`; `hidden-xs-up`/`hidden-md-down`/`hidden-lg-up` (стиль BS3); `pull-right`
  (в BS4 заменён на `float-right`); `class="column"` (`14.html`) — используется автором как
  обёртка внутри `row`/`no-gutters` **вместо** настоящего `col`/`col-*`, но `column` — не
  существующий класс Bootstrap 4, эффекта у него, скорее всего, нет вообще, а видимая
  раскладка получается просто за счёт `display:flex` у родительского `.row`. В новых
  шаблонах лучше использовать современные BS4-эквиваленты (`col`, `col-auto` и т.п.), а не
  копировать `column`.

## 5. Инлайн-CSS приёмы, подтверждённые в реальных дизайнах

Всё нижеперечисленное встречается **только** через атрибут `style="..."` — отдельного
CSS-файла/блока нет.

- **Градиенты:** многоточечный `linear-gradient(angle, ...)` — для декоративных полос,
  фонов карточек, даже двойного `linear-gradient` для имитации "миллиметровки".
- **Тени/свечение/псевдо-обводка текста:**
  - `box-shadow` — как мягкая, так и характерная "жёсткая" без блюра
    (`box-shadow: 10px 10px #1561b5;`);
  - `text-shadow`, в т.ч. **несколько теней через запятую** для эффекта свечения;
  - `filter: drop-shadow(1px 1px c) drop-shadow(-1px -1px c) drop-shadow(-1px 1px c)
    drop-shadow(1px -1px c)` — стек из 4 `drop-shadow` в разные стороны имитирует обводку
    текста (`-webkit-text-stroke` не используется ни разу).
- **Блендинг:** `mix-blend-mode` и `background-blend-mode` (значения: multiply, screen,
  overlay, darken, lighten, difference, exclusion, hue, saturation, color, luminosity,
  hard-light, soft-light, color-dodge, color-burn). Один из авторов прямо предупреждает,
  что рендерится непредсказуемо в разных браузерах — использовать с осторожностью.
- **Фильтры:** `filter: grayscale(100%)`, а также `filter: opacity(40%)` (`13_folders.html`)
  — функциональный аналог свойства `opacity`, но через CSS-функцию `filter`; встречается
  реже прямого `opacity:`, оба варианта рабочие.
- **Трансформации:** `rotate()`, `skewX()` (декоративные "скошенные" плашки без картинок/
  clip-path), `scaleX()`/`scaleY()` (растяжение/зеркалирование текста и спрайтов, в т.ч.
  `scaleX(-1)` для отзеркаливания картинки), `translate()`.
- **`clip-path: polygon(...)`** — чистые CSS-треугольники для декоративных "уголков".
- **Слои и позиционирование:** `position: absolute/relative/sticky` (в т.ч. "прилипающие"
  при скролле боковые колонки `position: sticky; top: 0`), `z-index`, decor-иконки,
  вынесенные за край карточки отрицательными `left`/`right`/`margin`.
- **Скроллируемые панели:** фиксированная `height`/`max-height` + `overflow: auto`/
  `scroll` + `scrollbar-width: thin` и `scrollbar-color: <thumb> <track>` — это
  **Firefox-специфичное** API кастомизации скроллбара; `::-webkit-scrollbar`
  (Chrome/Safari) недоступен, т.к. `<style>`-блоков нет.
- **Изображения:** `object-fit`/`object-position` для обрезки без искажений (у одного
  автора даже комментарий-подсказка про разницу fill/contain/cover/scale-down/none).
- **CSS-переменные (custom properties) как система теминга** — самый продвинутый приём,
  встречен в нескольких больших шаблонах:
  ```html
  <div class="card" style="--accent:#C7B3FF; --accent2:#9ED4FF;">
    ...
    <div style="background: var(--accent);">...</div>
    <div style="background: linear-gradient(to bottom, var(--accent) 50%, var(--accent2) 50%);">...</div>
  </div>
  ```
  Переменные объявляются один раз на корневом элементе через `style="--name:value;"` и
  переиспользуются во вложенных элементах через `var(--name)` — удобно для быстрой смены
  палитры одним find-replace.
- **Типографика:** `font-variant: small-caps`, `letter-spacing`, `word-spacing`,
  `white-space: nowrap`, `text-decoration-style: dashed` (пунктирное подчёркивание вместо
  сплошного). `font-family` — безопасны системные стеки (`Arial Black`, `Impact`,
  `Andale Mono`, `Comic Sans MS`, `MS Gothic`, `Monaco`, `Calibri`, `Times, serif`); в паре
  шаблонов встречаются имена Google Fonts (`'Space Mono'`, `'Space Grotesk'`) **без
  какого-либо `@import`/подключения** — то есть сработает шрифт, только если он уже
  установлен у зрителя локально или подключён самим сайтом глобально; это
  **непроверенный/рискованный** приём, а не гарантированная возможность.
- **`opacity` как приём "спрятать, но оставить интерактивным"** — значения вроде
  `0.0001`/`0.001`/`0.01` используются не для визуальной прозрачности, а чтобы полностью
  визуально скрыть элемент (iframe, audio), сохранив его кликабельным/играющим.
- **`pointer-events: none`** — на декоративных наложенных слоях (текстуры, фильтры), чтобы
  они не перехватывали клики у контента под ними.
- **`calc()`** — например, `width: calc(100% - 40px)` для фонового "свечения" чуть меньше
  родителя.
- Встречается и **нестандартный/"свободный" синтаксис** `rgb(23,42,107, 0.5)` (4 значения
  через запятую вместо `rgba()`) — работает не во всех браузерах по спецификации, лучше
  всегда явно использовать `rgba()`.

## 6. Прочие теги и особенности

- `<table>` используется не для таблиц данных, а как **CSS-хак центрирования**:
  `<table style="width:0%"><tbody><tr><td style="width:100%; ...">...</td></tr></tbody></table>`
  — "сжаться по контенту + растянуть внутренний td на 100%".
- Юникод/эмодзи/иероглифы/декоративные символы (напр. `𓂋⠀˚⠀﹒`, `₊`, `꒱꒱`, японский,
  греческий текст) вставляются прямо в разметку без экранирования и корректно
  отображаются.
- `alt` у `<img>` практически никогда не используется авторами шаблонов (не значит, что
  не работает — просто не в ходу в этом сообществе).
- `id` должен быть **уникален на странице**, т.к. вкладки/аккордеон/карусель матчатся по
  `href="#id"` ↔ `id="id"`. При копировании повторяющихся блоков (например, треков
  плейлиста `song1`, `song2`...) авторы explicitly предупреждают синхронизировать номера
  в `id`/`data-target`/`data-parent`.
- `target="_blank"` используется очень редко (замечен только один раз) — по умолчанию
  ссылки открываются в том же окне.
- **Реальная доступная ширина контента не такая уж и маленькая.** Компактные
  бейджи/карточки (`2.html`, `3.html`, `6.html` и т.п.) сами по себе узкие
  (`max-width` порядка 250–650px), но это выбор дизайна, а не ограничение площадки:
  `11.html` и `12.html` (desktop-тема) используют внешний контейнер
  `style="max-width: 1000px"` с многоколоночной раскладкой внутри — то есть Toyhouse
  реально рендерит и достаточно широкие/сложные многоколоночные лэйауты, а не только
  узкие виджеты. При проектировании широких дизайнов (например, с постоянной боковой
  колонкой меню) можно закладываться на ширину порядка 1000px, а не только на ~300–650px.
- Полностью валиден `<b>` наравне с `<strong>`, `<i>` наравне с `<u>` для декоративных
  элементов (в некоторых шаблонах `<i>` — это именно иконка Font Awesome, а не курсив,
  различай по классу: `class="fas fa-..."` = иконка, без класса = курсив).
- `<mark>` — подтверждён рабочий тег для "подсвеченного" текста (`14.html`):
  `<mark style="background-color:#7589c1">highlighted</mark>` — цвет подсветки задаётся
  инлайновым `style`, а не браузерным дефолтным жёлтым.
- **Ещё больше подтверждений "нестрогого парсера"** (см. раздел 0), теперь от разных
  независимых авторов: классы с обрубленным суффиксом типа `border-` или `bg-` (вместо
  `border-0`/`bg-primary`) встречаются в нескольких шаблонах и просто ничего не делают,
  без ошибок; CSS-объявления с пропущенным двоеточием (`height 500px;`) или вовсе без
  значения (`background-color;`, `overflow-auto;` — тут явно спутали Bootstrap-класс
  `overflow-auto` с CSS-свойством и по ошибке вставили прямо в `style=""`) — тоже просто
  молча игнорируются браузером, а не ломают всю страницу. Вывод для генерации новых
  шаблонов: опечатка не критична, но и не сработает — важно вычитывать сгенерированный код.

## 7. Конвенции сообщества (не техническое ограничение, но важно учитывать)

- **Плейсхолдеры** пишут заглавными буквами с очевидным именем:
  `URLHERE`, `LINKHERE`, `LINK_HERE`, `MP3LINK`, `IMAGEURL`, `VIDEOID`,
  `YOUTUBE_VIDEO_ID_HERE`. Если генерируешь новый шаблон — придерживайся той же
  конвенции, это ожидаемо пользователями сайта.
- Комментарии `<!-- -->` внутри шаблона — это **инструкции для человека, который будет
  редактировать код** (обычно живут только в исходнике/на внешних редакторах), сам сайт
  их при сохранении вырезает. Не полагайся на то, что комментарий останется виден на
  живом профиле.
- Почти в каждом шаблоне есть **обязательный неудаляемый кредит** автору кода (ссылка/
  тултип "Code by ..."), часто с явной пометкой `DO NOT REMOVE`. Это социальная норма
  сообщества авторов дизайнов, а не техническое ограничение сайта — но её стоит уважать
  при переиспользовании/адаптации чужого шаблона и предусматривать при создании нового.
- Перед вставкой кода в реальное поле на сайте — выключить WYSIWYG-режим (см. «КРИТИЧНО
  #1» в начале документа за подробным разбором, что именно ломается, если забыть).

## 8. Не встречалось ни разу → не считать гарантированно рабочим

`<canvas>`, `<svg>`, `<marquee>`, `<details>`/`<summary>`, чекбокс/радио-хаки для вкладок,
`<form>`/`<input>`, `autoplay`/старт-громкость/плейлисты на `<audio>`/`<video>` (по
свидетельству самих шаблонов — блокируются площадкой), свои `@font-face`-шрифты, любой
JS-обработчик событий. Если нужна такая функциональность — она, скорее всего, не будет
работать на toyhou.se; проверяй вручную перед тем как закладывать в дизайн.

## 10. Итог одной строкой

Дизайны toyhou.se — это **чистый HTML-фрагмент + инлайновые `style=""` + классы уже
подключённых на сайте Bootstrap 4 / Font Awesome / (Tooltipster)**. Никакого своего CSS
(`<style>`), JS (`<script>`) или кастомных шрифтов/анимаций (`@font-face`/`@keyframes`) —
нельзя. Всё "динамическое" (вкладки, аккордеон, карусель, тултипы) держится на
`data-toggle`/`data-target`/`data-parent`-атрибутах, которые обрабатывает уже
подключённый на сайте Bootstrap JS.
